## 一、面试主线

```text
单库本地事务
  -> 服务拆分 / 数据库拆分
  -> 一个业务操作跨多个资源
  -> 强一致性还是最终一致性
  -> Seata / MQ / TCC / Saga
  -> 分库分表后的查询、事务、ID、扩容问题
```

4 年经验要能说清楚：**不是所有分布式一致性问题都应该上分布式事务，很多时候最终一致性和业务补偿更稳**。

## 二、为什么会有分布式事务

本地事务只保证同一个数据库连接里的操作原子性：

```text
begin
  update account
  insert order
commit
```

当业务拆成多个服务或多个库：

```text
订单服务创建订单
  -> 库存服务扣库存
  -> 账户服务扣余额
  -> 优惠券服务核销券
```

任何一步失败都可能产生不一致。

典型问题：

1. 订单成功但库存没扣。
2. 库存扣了但订单失败。
3. 余额扣了但消息发送失败。
4. 网络超时不知道对方是否执行成功。

## 三、常见一致性方案

| 方案 | 一致性 | 优点 | 缺点 | 场景 |
| --- | --- | --- | --- | --- |
| 2PC/XA | 强一致 | 模型清晰 | 阻塞、性能差、资源锁定长 | 少量强一致场景 |
| Seata AT | 近似无侵入 | 接入相对简单 | 依赖 undo_log，SQL 有限制 | 普通关系型数据库事务 |
| TCC | 强业务控制 | 性能和可控性好 | 业务侵入大 | 资金、库存等核心链路 |
| Saga | 最终一致 | 适合长事务 | 补偿复杂 | 跨系统长流程 |
| 可靠消息 | 最终一致 | 解耦、吞吐好 | 有延迟，需要补偿 | 订单后置通知、积分、消息 |
| 本地消息表 | 最终一致 | 简单可靠 | 需要轮询/清理 | 中小系统常用 |

工程判断：

1. 资金类更偏强一致或 TCC。
2. 通知、积分、日志、搜索索引更适合最终一致。
3. 低价值链路不要为了“完美一致”引入过高复杂度。

## 四、Seata 核心机制

### 4.1 三个角色

| 角色 | 含义 |
| --- | --- |
| TC | Transaction Coordinator，全局事务协调器 |
| TM | Transaction Manager，全局事务发起方 |
| RM | Resource Manager，资源管理器，管理分支事务 |

流程：

```text
TM 开启全局事务
  -> RM 注册分支事务
  -> 各服务执行本地事务
  -> TM 通知 TC 提交或回滚
  -> TC 协调各 RM 完成二阶段
```

### 4.2 AT 模式做了什么

AT 模式一阶段：

1. 解析 SQL。
2. 查询修改前数据，生成 before image。
3. 执行业务 SQL。
4. 查询修改后数据，生成 after image。
5. 写入 `undo_log`。
6. 提交本地事务。

二阶段提交：

1. 全局成功：异步删除 `undo_log`。
2. 全局失败：根据 `undo_log` 生成反向 SQL 回滚。

### 4.3 AT 模式的限制

1. 依赖支持本地 ACID 的关系型数据库。
2. SQL 类型有支持边界。
3. 大事务会带来 undo_log 膨胀。
4. 长事务会增加锁竞争。
5. 脏写冲突需要特别关注。

## 五、AT、TCC、Saga 怎么选

### 5.1 AT

适合：

1. 传统 CRUD。
2. 业务不想大量改造。
3. 数据库操作相对简单。

不适合：

1. 非关系型资源。
2. 复杂 SQL。
3. 高并发热点扣减。
4. 长事务。

### 5.2 TCC

TCC 分三步：

```text
Try: 资源预留
Confirm: 确认提交
Cancel: 取消释放
```

例子：冻结余额。

```text
Try: 冻结 100 元
Confirm: 冻结金额转为已扣减
Cancel: 解冻 100 元
```

TCC 必须处理：

1. 幂等。
2. 空回滚。
3. 悬挂。
4. 防重。

### 5.3 Saga

Saga 把长事务拆成多个本地事务，每个事务有对应补偿动作。

```text
创建订单
  -> 扣库存
  -> 扣余额
  -> 失败后补偿：退余额、还库存、取消订单
```

适合审批流、跨外部系统、长时间业务流程。

## 六、可靠消息最终一致性

### 6.1 本地事务 + 消息发送的问题

```text
先更新数据库，再发 MQ
```

如果数据库提交成功后，发送 MQ 失败，就会不一致。

### 6.2 本地消息表

流程：

```text
本地事务内：
  -> 写业务数据
  -> 写消息表 status=NEW
提交后：
  -> 定时任务扫描 NEW 消息
  -> 投递 MQ
  -> 成功后标记 SENT
消费端：
  -> 幂等消费
```

优点是简单可靠。缺点是有延迟，需要消息表清理和补偿任务。

### 6.3 MQ 事务消息

事务消息通常流程：

```text
发送半消息
  -> 执行本地事务
  -> 提交或回滚消息
  -> MQ 回查未确认事务状态
```

消费者仍然必须幂等，因为消息可能重复投递。

## 七、分库分表基础

### 7.1 为什么分库分表

常见原因：

1. 单表数据太大，索引层级变高，查询变慢。
2. 单库连接、CPU、IO、存储压力大。
3. 写入吞吐不足。
4. 单库故障影响面太大。

不是所有大表都要分表。先考虑：

1. 索引优化。
2. SQL 改写。
3. 冷热数据归档。
4. 读写分离。
5. 缓存。
6. 垂直拆表。

### 7.2 垂直拆分和水平拆分

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| 垂直分库 | 按业务域拆库 | 用户库、订单库、商品库 |
| 垂直分表 | 拆宽表字段 | 用户基础表、用户扩展表 |
| 水平分库 | 同一表按规则分到多个库 | 订单按 user_id 分库 |
| 水平分表 | 同一表按规则分成多张表 | order_00 到 order_15 |

### 7.3 分片键怎么选

好的分片键：

1. 查询频率高。
2. 分布均匀。
3. 不经常变更。
4. 能减少跨分片查询。
5. 与业务归属一致。

订单系统常见选择：

1. 买家端按 `user_id`。
2. 商家端按 `shop_id`。
3. 查询主要按订单号时，用订单号里嵌入分片信息。

分片键选错，后面很多查询都会跨库扫。

## 八、ShardingSphere 核心概念

### 8.1 它解决什么问题

ShardingSphere 可以在应用侧提供：

1. 分库分表路由。
2. 读写分离。
3. 分布式主键。
4. 分布式事务集成。
5. SQL 改写和结果归并。

基本流程：

```text
应用发 SQL
  -> SQL 解析
  -> 路由到真实库表
  -> SQL 改写
  -> 执行
  -> 结果归并
```

### 8.2 路由类型

| 类型 | 含义 |
| --- | --- |
| 精确路由 | 分片键等值查询，能定位到一个或少数分片 |
| 范围路由 | 按范围查询，可能路由到多个分片 |
| 全路由 | 没有分片键，扫描所有分片 |
| 广播表 | 每个库都有一份，如字典表 |
| 绑定表 | 分片规则一致，可避免笛卡尔路由 |

### 8.3 分库分表后的 SQL 限制

容易变复杂的查询：

1. 跨分片 join。
2. 跨分片 group by。
3. 跨分片 order by + limit。
4. 非分片键查询。
5. 分布式事务。
6. 全局唯一约束。

工程建议：

1. 高频查询必须带分片键。
2. 尽量避免跨库 join，在应用层组装或做冗余字段。
3. 报表统计走离线数仓或专门宽表。
4. 管理后台模糊查询要控制范围和权限。

## 九、分布式 ID 与全局唯一性

分库分表后，自增 id 容易冲突，需要全局 ID。

常见方案：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| UUID | 简单、本地生成 | 太长、无序、索引不友好 |
| Redis 自增 | 简单递增 | 依赖 Redis，高可用要处理 |
| 数据库号段 | 趋势递增，性能好 | 需要号段服务 |
| Snowflake | 本地生成，趋势递增 | 时钟回拨问题 |

Snowflake 结构通常包括：

```text
时间戳 + 机器号 + 序列号
```

注意：

1. 机器号唯一。
2. 时钟回拨保护。
3. 前端 Long 精度问题。
4. ID 中是否要嵌入分片信息。

## 十、常见线上问题和排查

### 10.1 Seata 回滚失败

排查：

```text
全局事务状态
  -> 分支事务状态
  -> undo_log 是否存在
  -> before/after image 是否匹配
  -> 是否发生脏写
  -> 业务 SQL 是否被支持
  -> TC、RM 日志
```

### 10.2 分片查询很慢

排查：

```text
SQL 是否带分片键
  -> 是否全路由
  -> 每个真实 SQL 是否走索引
  -> 结果归并数据量
  -> order by / group by / limit 是否跨分片
  -> 是否需要改查询模型
```

### 10.3 数据倾斜

原因：

1. 分片键分布不均。
2. 大客户或热点用户集中。
3. 按时间分片导致新表热点。

处理：

1. 调整分片算法。
2. 热点 key 单独拆分。
3. 扩容迁移。
4. 读写分离和缓存。
5. 业务限流削峰。

### 10.4 扩容迁移

基本步骤：

```text
设计新分片规则
  -> 双写或同步历史数据
  -> 校验数据一致性
  -> 灰度读新库
  -> 切流量
  -> 保留回滚窗口
  -> 清理旧数据
```

扩容最怕没有回滚方案。切换前必须有校验、监控和降级。

## 十一、面试高频回答模板

### 11.1 分布式事务怎么选型

> 我会先判断业务是否真的需要强一致。资金、库存这类核心链路可以考虑 TCC 或 Seata，但要接受复杂度和性能成本。通知、积分、搜索索引、报表同步这类可以用可靠消息最终一致性。本地消息表是比较稳的方案：业务数据和消息记录在同一个本地事务里提交，再由任务投递 MQ，消费端做幂等和补偿。

### 11.2 Seata AT 模式原理

> Seata AT 模式一阶段会解析 SQL，记录修改前后的镜像到 undo_log，并提交本地事务；二阶段如果全局提交，就异步清理 undo_log，如果全局回滚，就根据 undo_log 生成反向 SQL 回滚。它对业务侵入较低，但依赖数据库本地事务和 SQL 支持范围，不适合长事务、高并发热点和特别复杂的 SQL。

### 11.3 TCC 要解决哪些问题

> TCC 把业务拆成 Try、Confirm、Cancel。Try 做资源预留，Confirm 做确认提交，Cancel 做释放。它的难点是幂等、空回滚和悬挂。比如 Cancel 可能先于 Try 到达，就要能识别空回滚；Confirm 或 Cancel 可能重复调用，所以必须通过事务状态表或唯一约束保证幂等。

### 11.4 分库分表分片键怎么选

> 分片键要优先选择高频查询条件、分布均匀、不易变更、能减少跨分片查询的字段。订单系统如果买家端查询为主，可以按 user_id；如果商家端查询为主，可能按 shop_id；如果订单号查询最多，可以在订单号里嵌入分片信息。分片键选错，会导致大量全路由和跨库聚合，后期成本很高。

### 11.5 分库分表后跨库查询怎么办

> 首先尽量避免跨库 join，把查询模型设计成带分片键的单分片查询。必要时可以做字段冗余、应用层二次查询组装、广播表、绑定表，或者把统计报表放到 ES、数仓、宽表里。后台管理的模糊查询要控制时间范围和条件，不能让它随意全分片扫描。
