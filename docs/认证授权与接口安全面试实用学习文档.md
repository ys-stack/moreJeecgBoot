## 一、认证授权面试主线

后台系统安全的面试链路通常是：

```text
用户登录
  -> 认证身份
  -> 生成 token / session
  -> 每次请求解析身份
  -> 判断角色、菜单、按钮、接口权限
  -> 追加数据权限
  -> 记录审计日志
  -> 防止越权、重放、注入、XSS
```

面试官真正关心的是：**你是否知道权限不是一个注解就完事，而是一套从入口到数据层的防线**。

## 二、认证、授权、鉴权、会话

| 概念 | 含义 | 例子 |
| --- | --- | --- |
| 认证 Authentication | 证明你是谁 | 用户名密码、短信验证码、OAuth 登录 |
| 授权 Authorization | 给你什么权限 | 给用户分配角色、菜单、按钮 |
| 鉴权 Access Control | 每次访问时判断能不能访问 | 请求 `/sys/user/list` 前检查权限 |
| 会话 Session | 保存登录状态 | 服务端 Session、Redis Session、JWT |

一句话：

> 认证解决身份问题，授权解决权限分配问题，鉴权解决访问时的权限判断问题，会话解决登录状态延续问题。

## 三、JWT 登录机制

### 3.1 JWT 结构

JWT 通常由三段组成：

```text
Header.Payload.Signature
```

| 部分 | 内容 |
| --- | --- |
| Header | token 类型、签名算法 |
| Payload | 用户 id、租户、过期时间、权限版本等声明 |
| Signature | 用密钥对前两段签名，防篡改 |

JWT 的重点是：**默认只签名，不加密**。Payload 里的内容可以被客户端解码看到，所以不要放密码、手机号明文、身份证号等敏感信息。

### 3.2 JWT 登录流程

```text
用户名密码登录
  -> 校验密码和账号状态
  -> 查询用户角色、租户、权限版本
  -> 生成 access token
  -> 前端保存 token
  -> 请求时放入 Header
  -> 后端过滤器解析 token
  -> 构建当前用户上下文
  -> 执行接口权限和数据权限判断
```

### 3.3 JWT 优点和缺点

| 优点 | 缺点 |
| --- | --- |
| 无状态，服务端扩展简单 | 服务端主动失效困难 |
| 适合前后端分离和微服务 | token 泄露后在过期前可被使用 |
| 可携带少量身份声明 | payload 过大会增加请求体积 |
| 不依赖服务端 session | 权限变更后可能有延迟 |

### 3.4 怎么让 JWT 失效

常见方案：

1. access token 短过期时间 + refresh token 续期。
2. Redis 维护 token 黑名单。
3. 用户表维护 `tokenVersion` 或 `passwordUpdateTime`，解析 token 后比对。
4. 权限变更、踢人下线时删除服务端缓存或提升版本号。

工程取舍：

> 如果系统要求强制下线和权限实时生效，JWT 不能完全无状态，通常会配合 Redis 或权限版本校验。

## 四、Shiro 核心机制

### 4.1 Shiro 的核心组件

| 组件 | 职责 |
| --- | --- |
| `Subject` | 当前访问主体，代表当前用户或系统调用方 |
| `SecurityManager` | Shiro 核心调度器 |
| `Realm` | 连接业务数据，负责认证和授权数据查询 |
| `SessionManager` | 会话管理 |
| `CacheManager` | 缓存认证授权信息 |
| `Filter` | Web 请求入口鉴权 |

### 4.2 Shiro 认证流程

```text
Subject.login(token)
  -> SecurityManager
  -> Authenticator
  -> Realm.doGetAuthenticationInfo()
  -> 密码/凭证匹配
  -> 登录成功，保存身份
```

### 4.3 Shiro 授权流程

```text
请求进入过滤器
  -> 获取 Subject
  -> 判断角色或权限字符串
  -> 缓存中有授权信息则直接用
  -> 缓存没有则调用 Realm 查询
  -> 允许访问或拒绝访问
```

### 4.4 Shiro 和 Spring Security 怎么比较

| 对比项 | Shiro | Spring Security |
| --- | --- | --- |
| 学习成本 | 相对轻量 | 体系更完整，概念更多 |
| Web 安全能力 | 常见后台够用 | 与 Spring 生态结合更深 |
| OAuth2/OIDC | 需要额外整合 | 生态更成熟 |
| 适合场景 | 传统后台、权限模型可控 | Spring 全家桶、复杂认证授权 |

回答不要贬低任一方。可以说：

> Shiro 更轻量，适合很多后台管理系统；Spring Security 体系更完整，尤其在 OAuth2、资源服务器、方法安全上生态更强。项目选型要看团队历史、现有封装和安全复杂度。

## 五、RBAC、按钮权限与数据权限

### 5.1 RBAC 模型

最常见的 RBAC：

```text
用户
  -> 用户角色关系
  -> 角色
  -> 角色权限关系
  -> 菜单 / 按钮 / 接口权限
```

典型表：

| 表 | 含义 |
| --- | --- |
| `sys_user` | 用户 |
| `sys_role` | 角色 |
| `sys_user_role` | 用户角色关系 |
| `sys_permission` | 菜单、按钮、接口权限 |
| `sys_role_permission` | 角色权限关系 |

### 5.2 菜单权限、按钮权限、接口权限

| 权限 | 控制点 |
| --- | --- |
| 菜单权限 | 前端路由和菜单是否展示 |
| 按钮权限 | 页面按钮是否展示或可点击 |
| 接口权限 | 后端接口是否允许访问 |

注意：**前端隐藏按钮不是安全控制，只是用户体验**。真正的权限必须在后端接口层校验。

### 5.3 数据权限

数据权限解决“能访问接口，但只能看部分数据”的问题。

常见维度：

1. 本人数据。
2. 本部门数据。
3. 本部门及下级部门数据。
4. 指定部门数据。
5. 租户数据。
6. 自定义 SQL 规则。

常见实现方式：

1. 在 Mapper SQL 中手写条件。
2. AOP + 注解解析数据权限。
3. MyBatis 拦截器动态追加条件。
4. 数据库视图或行级安全策略。

工程注意：

1. 数据权限条件必须和业务查询别名匹配。
2. 分页 count SQL 也要带权限条件。
3. 导出接口、详情接口、批量接口也要检查。
4. 管理员绕过规则要明确。

## 六、接口安全常见方案

### 6.1 API 签名

适合开放接口、第三方对接、服务间调用。

常见参数：

```text
appKey
timestamp
nonce
bodyHash
signature
```

服务端校验：

1. 校验 appKey 是否存在。
2. 校验时间戳是否在允许窗口内。
3. 校验 nonce 是否已使用，防重放。
4. 按约定算法重算签名。
5. 比对签名。

### 6.2 幂等控制

常见方案：

1. 前端防重复提交。
2. 请求幂等号，如 `Idempotency-Key`。
3. Redis `SET NX EX`。
4. 数据库唯一索引。
5. 状态机校验。

面试回答重点：

> 前端防抖只能改善体验，真正的幂等必须在服务端。核心交易类接口通常用唯一业务号或幂等号，加数据库唯一约束兜底。

### 6.3 接口限流

常见维度：

1. IP。
2. 用户。
3. appKey。
4. 接口路径。
5. 租户。

常见算法：

1. 固定窗口。
2. 滑动窗口。
3. 漏桶。
4. 令牌桶。

分布式场景可以用 Redis、Sentinel、网关限流或 Nginx。

## 七、常见攻击与防护

### 7.1 SQL 注入

风险示例：

```java
String sql = "select * from sys_user where username = '" + username + "'";
```

防护：

1. 使用 MyBatis `#{}` 参数绑定，不用 `${}` 拼用户输入。
2. 动态排序字段使用白名单。
3. 高风险查询加参数校验。
4. 不把数据库错误详情返回前端。

### 7.2 XSS

XSS 是把恶意脚本注入页面。

防护：

1. 输入校验。
2. 输出编码。
3. 富文本白名单过滤。
4. 设置 CSP。
5. Cookie 设置 `HttpOnly`、`Secure`、`SameSite`。

后台系统尤其要注意公告、富文本、配置项、低代码表单里的 HTML 输入。

### 7.3 CSRF

CSRF 利用用户已登录状态发起伪造请求。

防护：

1. SameSite Cookie。
2. CSRF Token。
3. 关键操作二次确认。
4. 使用 Authorization Header 的 token 机制可降低传统 Cookie CSRF 风险。

### 7.4 文件上传安全

注意点：

1. 限制文件大小。
2. 校验后缀和真实 MIME。
3. 文件名重新生成，避免路径穿越。
4. 上传目录不允许执行脚本。
5. 图片可做二次处理。
6. 私有文件下载要鉴权。

## 八、后台系统常见越权问题

### 8.1 水平越权

用户 A 访问用户 B 的数据：

```text
/api/order/detail?id=1001
```

如果只判断“已登录”，没有判断订单是否属于当前用户或当前租户，就会水平越权。

防护：

1. 查询条件带当前用户、部门、租户。
2. 详情、编辑、删除接口都做归属校验。
3. 不信任前端传来的 tenantId、userId。

### 8.2 垂直越权

普通用户访问管理员接口：

```text
/sys/user/delete
```

防护：

1. 后端接口权限注解。
2. URL 级过滤规则。
3. 角色权限缓存及时刷新。
4. 管理端和开放端隔离。

### 8.3 多租户越权

多租户系统必须保证：

1. 当前租户上下文来源可信。
2. SQL 自动追加租户条件。
3. 跨租户管理能力有明确白名单。
4. 缓存 key 带租户维度。
5. 文件、消息、搜索索引也要隔离租户。

## 九、线上排查与安全兜底

### 9.1 登录失败排查

```text
账号是否存在
  -> 密码加密算法是否一致
  -> 账号是否锁定/禁用
  -> 验证码是否过期
  -> token 是否生成
  -> Redis/session 是否正常
  -> 前端 Header 是否携带 token
```

### 9.2 权限不生效排查

```text
用户角色是否正确
  -> 角色权限关系是否正确
  -> 权限标识是否和接口一致
  -> 授权缓存是否刷新
  -> 是否命中超级管理员逻辑
  -> 前端菜单是否重新拉取
```

### 9.3 安全兜底清单

1. 默认拒绝，显式放行。
2. 所有写操作做后端鉴权。
3. 详情、导出、批量接口同样做数据权限。
4. 敏感操作记录审计日志。
5. token、密钥、密码不打日志。
6. 密码使用强哈希和盐。
7. OpenAPI 使用签名、时间戳、nonce。
8. 生产环境关闭调试接口和详细错误堆栈。

## 十、面试高频回答模板

### 10.1 JWT 和 Session 怎么选

> Session 是服务端保存登录状态，控制能力强，比如踢人下线、状态变更实时生效都比较自然，但分布式场景要引入 Redis 共享 Session。JWT 是客户端携带身份凭证，服务端主要做签名校验，扩展性好，适合前后端分离和微服务，但主动失效困难。实际项目里如果要求强制下线和权限实时刷新，JWT 通常也会配合 Redis 黑名单、短 token 或权限版本号。

### 10.2 Shiro 的认证授权流程

> Shiro 里当前用户抽象成 Subject，核心调度由 SecurityManager 完成，Realm 对接业务数据。登录时 Subject 提交 token，Realm 查询认证信息并完成凭证匹配；访问接口时过滤器或注解触发鉴权，Shiro 会从缓存或 Realm 获取角色权限信息，然后判断是否允许访问。项目里还会结合 JWT，让请求入口先解析 token，再构建当前用户身份。

### 10.3 RBAC 怎么设计

> RBAC 的核心是用户关联角色，角色关联权限。权限可以细分为菜单、按钮和接口。菜单和按钮主要控制前端展示，接口权限必须在后端校验。对于后台系统，还要在接口权限之外加数据权限，比如本人、本部门、本租户数据，否则用户虽然只能访问被授权接口，仍可能通过改参数看到不属于自己的数据。

### 10.4 怎么防止越权

> 防越权不能只靠前端。垂直越权靠后端接口权限控制，水平越权靠数据归属校验和数据权限条件。比如详情、编辑、删除、导出接口都要带当前用户、部门或租户条件，不能只根据前端传来的 id 查询。多租户系统还要保证缓存、文件、消息、搜索索引里的 key 或查询条件都带租户隔离。

### 10.5 开放接口怎么做安全

> 开放接口通常用 appKey/appSecret 做签名。请求携带时间戳、随机数、请求体摘要和签名，服务端根据 appKey 找到密钥，校验时间窗口、防重放 nonce、重算签名并比对。关键写接口还要做幂等控制和限流，避免重复调用、恶意刷接口或重放攻击。
